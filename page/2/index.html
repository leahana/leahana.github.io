Markdown本地笔记 Obsidian

需求：需要一个笔记存放提示词学习过程中的提示词
思考：问chat工具 有哪些工具，挑选一个工具，再次询问，使用5W1H分析是否适合自己。

Obsidian 简介（基于5W1H框架）

1. What（是什么？）

  • 定位：基于Markdown的本地优先笔记软件
  • 核心功能：
    • 双向链接（双向引用笔记）
    • 知识图谱（可视化笔记关联）
    • 插件生态（900+扩展功能）
  • 典型用途：知识管理、写作、构建第二大脑

2. Who（目标用户？）

  • 🧠 知识密集型工作者：学者、作家、研究员
  • 🛠️ 技术爱好者：偏好数据本地化、支持Markdown和开源插件的用户
  • 🧩 个性化需求者：需高度自定义笔记流程的人群

3. When（何时推出？）

  • 发布时间：2020年
  • 迭代节奏：持续更新，已推出移动端、Canvas白板等功能

4. Where（适用场景？）

  • 设备支持：
    • 桌面端：Windows/macOS/Linux
    • 移动端：iOS/Android
  • 存储模式：
    • 默认本地存储（可搭配iCloud/Dropbox同步）
    • 支持端到端加密的付费云服务（Obsidian Sync）

5. Why（核心优势？）

  • ✅ 数据主权：完全掌控本地文件（存储为.md格式）
  • ✅ 网状思维：通过链接实现非线性的知识连接
  • ✅ 零成本启动：基础功能免费，无强制订阅
  • ✅ 隐私保护：无广告、无数据上传分析

6. How（如何使用？）

  1. 基础操作：
    • 用[[ ]]语法创建笔记链接
    • 通过Ctrl/Cmd + O快速跳转笔记
  2. 进阶功能：
    • 插件市场安装工具（如日记模板、看板管理）
    • 用CSS自定义主题样式
  3. 协作场景：
    • 通过Git版本控制管理笔记
    • 使用付费版Obsidian Sync实时同步

同类工具对比

工具 优势 局限性
Roam Research 实时协作/块级引用 纯云端/价格高($15/月)
Logseq 开源免费/大纲笔记 移动端体验差
Notion 数据库强大/团队协作 依赖网络/导出困难
RemNote 内置记忆卡片系统 界面复杂/学习成本高
TiddlyWiki 单文件便携/完全开源 多设备同步需技术配置

选择建议

❗ 选Obsidian如果：

  • 需要完全控制数据存储位置
  • 习惯Markdown原生语法
  • 愿意通过插件逐步搭建工作流

❗ 考虑替代品当：

  • 需要开箱即用协作 → Notion
  • 专注大纲笔记 → Logseq
  • 研究学术文献 → Roam Research

Factory Method 模式

Factory Method 模式

▎模式结构

框架和具体加工

用相同的框架创建出其他的“产品”和“工厂”。例如，我们这次要创建表示电视机类的Television和表示电视机工厂的TelevisionFactory 。这里我们只需要impor framework包就可以编写television包。我们这里没有修改， 根本没有必要修改framwork的内容就可以创建出其他的“产品”和“工厂”。framework不依赖于idcard包，television包。

生成实例—方法的三种实现方式

指定其为抽象方法

​ 一旦将createProduct指定为抽象方法后， 子类就必须实现该方法。如果子类不实现该方法，编译器将会报告编译错误，也就是示例程序所采用的方式。

为其实现默认处理

​ 实现默认处理后，如果子类没有实现该方法，将进行默认处理。这时是使用new 关键字创建出实例的，因此不能将Product 类定义为抽象类。

在其中抛出异常

​ createProduct方法的默认处理为抛出异常，这样一来，如果未在子类实现中实现该方法，程序运行就会报错（报错，告知开发人员没有实现createProduct 方法）, 不过需要另外编写FactoryMethodRuntimeException

Builder 模式

Builder模式

组装复杂的实例

Builder模式中登场的角色

Builder （建造者）

负责定义用于生成实例的接口

ConcreteBuilder（具体的建造者）

实现Builder角色的类，定义了在生成实例时实际被调用的方法。

Director（建工）

负责使用Builder角色的接口来生成实例，并不依赖于ConcreteBuilder角色。为了确保无论ConcreteBuilder如何被定义，Director都能正常工作，他只调用在builder角色中被定义的方法。

相关设计模式：

  • Template Method 模式
  • Composite 模式
  • Abstract Factory 模式
  • Facade 模式

拓展思路：

在面向对象编程中“谁知道什么”非常重要。我们需要在编程时注意哪个类可以使用哪个方法，以及用这个方法好不好。Director不知道自己使用的究竟是Builder的哪个子类，这样才能实现可替换性。

Singleton 模式

Singleton模式

Singleton 中登场的角色

Singleton

在Singleton模式中，只有Singleton这一个角色，Singleton角色中有一个返回实例的static 方法，该方法总会返回同一个实例

拓展思路的要点

  • 为什么必须设置限制

存在多个实例时，实例之间相互影响，可能产生意想不到的bug，确保只有一个实例的时候，就可以在这个前提条件下放心编程，并且不会造成资源浪费的情况发生。不用每次使用某个对象而一直创建他的实例。

  • 何时生成唯一实例

程序运行后，第一次调用getInstance方法是，Singleton类就会被初始化，也就是在这时候，static修饰的singleton被初始化，生成了唯一一个实例

相关的设计模式

  • AbstractFactory模式
  • Builder模式
  • Facade模式
  • Prototype模式

ConcretePrototype（具体的原型）

Client（使用者）

负责使用复制实例的方法生成新的实例

MCP 模型上下文协议

MCP 模型上下文协议

1. MCP 简介

模型上下文协议（MCP）是一种标准化协议，使 AI 模型能够以结构化方式与外部工具和资源交互。它支持多种传输机制，以便在不同环境中提供灵活性。

1.1 MCP 本质

  • 标准化协议实现AI模型与外设的结构化交互
  • 类似Type-C接口的function call统一标准
  • mcp-全屋智能家电标准，function call-开关/遥控器

2. MCP 实践

2.1 Java MCP 实现

Java MCP 实现遵循三层架构：

2.1.1 架构层次

  • 客户端/服务器层：McpClient 处理客户端操作，而 McpServer 管理服务器端协议操作。两者都使用 McpSession 进行通信管理。
  • 会话层（McpSession）：通过 DefaultMcpSession 实现管理通信模式和状态。
  • 传输层（McpTransport）：处理 JSON-RPC 消息序列化和反序列化，并支持多种传输实现。

2.1.2 MCP 客户端

MCP 客户端是模型上下文协议（MCP）架构中的关键组件，负责建立和管理与 MCP 服务器的连接。它实现协议的客户端，处理：

  • 协议版本协商，确保与服务器兼容
  • 用于确定可用功能的功能协商
  • 消息传输和 JSON-RPC 通信
  • 工具发现和执行
  • 资源访问和管理
  • 提示系统交互
可选功能
  • 根管理
  • 采样支持
  • 同步和异步操作
传输方式
  • 基于 Stdio 的传输，用于基于进程的通信
  • 基于 Java HttpClient 的 SSE 客户端传输
  • 用于反应式 HTTP 流的 WebFlux SSE 客户端传输

2.1.3 MCP 服务器

MCP 服务器是模型上下文协议（MCP）架构中的基础组件，用于为客户提供工具、资源和功能。它实现了协议的服务器端，负责：

服务器端协议操作实现
  • 工具曝光和发现
  • 使用基于 URI 的访问进行资源管理
  • 及时的模板配置和处理
  • 与客户的能力协商
  • 结构化日志记录和通知
并发管理
  • 并发客户端连接管理
  • 同步和异步 API 支持
传输实现
  • 基于 Stdio 的传输，用于基于进程的通信
  • 基于 Servlet 的 SSE 服务器传输
  • 用于反应式 HTTP 流的 WebFlux SSE 服务器传输
  • 用于基于 servlet 的 HTTP 流的 WebMVC SSE 服务器传输
